### Q1：Android 项目接入 Flutter 有哪些方式？FlutterActivity 和 FlutterFragment 如何选择？

- **难度：** ⭐⭐⭐
- **考查点：** Add-to-App 方案、混合开发架构

**参考答案：**

**三种接入方式：**

| 方式 | 适用场景 | 优缺点 |
|------|---------|--------|
| `FlutterActivity` | 整个屏幕都是 Flutter 内容 | 最简单，直接 startActivity；无法与原生 View 混排 |
| `FlutterFragment` | Flutter 嵌入原生 Activity 的一部分 | 灵活，可与原生 View 组合；需管理 Fragment 生命周期 |
| `FlutterView`（直接操作） | 极度自定义布局 | 灵活性最高；代码最复杂，需手动管理引擎生命周期 |

**选择原则：**
- 新页面全部是 Flutter → `FlutterActivity`，最简洁
- 需要原生 TabBar / NavigationBar 包裹 Flutter 内容 → `FlutterFragment`
- Flutter 只是页面中某个区域（如地图+Flutter表单混排）→ `FlutterView`

**加分点：** 提到 `FlutterEngineCache` 预热引擎，避免首次创建 Engine 的 300ms-1s 延迟。

### Q2：FlutterEngine 预热的原理和最佳实践是什么？

- **难度：** ⭐⭐⭐
- **考查点：** 混合开发性能优化

**参考答案：**

**问题根源：** 创建 `FlutterEngine` 会初始化 Dart VM、加载 Flutter Framework 库、执行 Dart 入口函数，首次耗时 300ms-1000ms，会造成明显的启动白屏。

**预热方案：**
```kotlin
// Application.onCreate() 中提前初始化（App 启动时异步执行）
class MyApp : Application() {
    override fun onCreate() {
        super.onCreate()
        // 在后台线程预热，不阻塞主线程
        Thread {
            FlutterEngine(this).also { engine ->
                engine.dartExecutor.executeDartEntrypoint(
                    DartExecutor.DartEntrypoint.createDefault()
                )
                FlutterEngineCache.getInstance().put("main_engine", engine)
            }
        }.start()
    }
}

// 使用预热的引擎（近乎 0 等待）
FlutterFragment.withCachedEngine("main_engine")
    .build<FlutterFragment>()
```

**多引擎策略：**
- 单引擎：适合单一 Flutter 模块，内存占用小（~4MB）
- 多引擎：不同功能模块相互隔离状态时使用，但每个引擎 4-6MB，不超过 3 个

**加分点：** 预热时机可结合 App 启动链路分析，选择在主线程空闲期（IdleHandler）执行，减少对启动性能的影响。

### Q3：混合开发中，原生和 Flutter 的路由栈如何统一管理？有哪些方案？

- **难度：** ⭐⭐⭐⭐⭐
- **考查点：** 混合路由架构、业界方案对比

**参考答案：**

**核心问题：** 原生维护 Activity 栈，Flutter 维护自己的 Navigator 栈，两栈独立管理导致返回手势混乱、内存中存在大量 Engine + Activity、生命周期混乱。

**方案对比：**

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 官方 Add-to-App（多 Activity） | 每个 Flutter 页面一个 FlutterActivity | 简单，官方支持 | 多引擎内存大，返回栈不统一 |
| **FlutterBoost（推荐）** | 单引擎 + 统一路由容器 | 内存小，跳转统一 | 侵入性强，升级有风险 |
| 自研 Mix Stack | 参考 FlutterBoost 思路定制 | 可按需定制 | 维护成本极高 |

**FlutterBoost 核心思路：**
```
所有页面跳转通过 BoostNavigator 统一调度
  原生跳 Flutter：BoostNavigator.open("flutter://home")
    → 复用 FlutterEngineGroup 内的单一 Engine
    → Flutter 内部 Navigator 跳到对应 route
  Flutter 跳原生：BoostNavigator.open("native://settings")
    → 通过 MethodChannel 通知原生 startActivity
  
路由栈统一维护：
  双端同步维护一份页面栈记录，返回时两端协同处理
```

**实际落地建议：**
1. 制定混合跳转规范：所有路由收口到 Router 模块，禁止 Feature 直接操作 Activity/Navigator
2. Flutter → 原生跳转一律走 MethodChannel，由原生统一 dispatch
3. NavigatorObserver 同步路由状态到埋点系统

### Q4：如何在混合项目中实现 Flutter 和原生共享登录态（Token）？

- **难度：** ⭐⭐⭐
- **考查点：** 混合通信、安全存储、架构解耦

**参考答案：**

**方案设计：**

```
原生持有 Token（单一数据源）
  ├── 原生页面：直接从 AccountManager / SharedPreferences 读取
  └── Flutter 页面：启动时通过 MethodChannel 获取 Token
        ↓
  Flutter 本地缓存（内存级别，不持久化）
        ↓
  Flutter 网络请求统一在 Interceptor 中注入 Token

Token 刷新流程：
  原生监听 Token 刷新 → 通知 Flutter（EventChannel）→ Flutter 更新内存缓存
```

**代码示例：**
```dart
// Flutter 端 - 启动时初始化
class AuthService {
  static const _channel = MethodChannel('auth/token');
  String? _token;

  Future<void> init() async {
    _token = await _channel.invokeMethod<String>('getToken');
  }

  // 监听 Token 刷新
  void listenTokenRefresh() {
    const EventChannel('auth/token_refresh')
        .receiveBroadcastStream()
        .cast<String>()
        .listen((newToken) => _token = newToken);
  }
}
```

**安全注意事项：**
- Token 不存储在 Flutter 本地持久化（`SharedPreferences` / `FlutterSecureStorage`），原生持有单一数据源
- 通过 Platform Channel 传输，不走不可信渠道
- 原生端 `FlutterSecureStorage` / `Keychain` 加密存储

### Q5：如何设计一个百万级 DAU App 的组件化架构？有哪些核心决策点？

- **难度：** ⭐⭐⭐⭐⭐
- **考查点：** 大型项目架构、工程治理、技术决策

**参考答案：**

**分层架构：**
```
app（壳工程）← 只负责组装，几乎无业务代码
├── feature 层（功能模块）
│   ├── feature_home
│   ├── feature_im
│   └── feature_payment
├── business 层（共享业务）
│   ├── biz_account（用户/登录）
│   └── biz_common（通用业务组件）
├── service 层（能力服务，接口定义）
│   ├── service_network
│   └── service_push
└── base 层（基础设施）
    ├── base_ui（通用 UI 组件库）
    ├── base_network（Retrofit 封装）
    └── base_utils（工具类）
```

**核心决策点：**

**1. 模块间通信策略**
- 页面跳转 → Navigation Component / ARouter（统一路由）
- 服务调用 → 接口下沉到 service 层，Hilt 注入实现
- 事件通知 → SharedFlow 事件总线（不引入直接依赖）

**2. 独立运行调试**
- 每个 feature 模块配置 `debugApp` flavor，可单独打包运行
- 提供 Mock 数据层，不依赖真实后端

**3. 构建速度优化**
- 模块化后配合 Gradle Build Cache，只编译变更模块
- 接入 ABI 拆包，减少 CI 打包时间

**4. 版本管理统一**
- 所有依赖版本集中在 `libs.versions.toml`（Version Catalog）
- 统一 `minSdk` / `compileSdk`，通过公共 Plugin 强制执行

### Q6：如何设计 App 的网络层架构，使其具备可替换、可测试、可降级的特性？

- **难度：** ⭐⭐⭐⭐
- **考查点：** 网络层架构、接口设计、可测试性

**参考答案：**

**分层设计：**
```
UI / ViewModel
    ↓
Repository（数据仓库）← 唯一数据入口，屏蔽数据来源
    ↓                ↓
RemoteDataSource   LocalDataSource
（Retrofit API）   （Room DAO）
    ↓
HttpClient（OkHttp 封装层）
    ├── AuthInterceptor（Token 注入 + 刷新）
    ├── RetryInterceptor（失败重试）
    ├── LoggingInterceptor（调试日志）
    └── CacheInterceptor（离线缓存）
```

**统一结果封装：**
```kotlin
sealed class Result<out T> {
    data class Success<T>(val data: T) : Result<T>()
    data class Error(val exception: AppException) : Result<Nothing>()
    data object Loading : Result<Nothing>()
}
```

**可替换（接口隔离）：**
```kotlin
interface NetworkClient {
    suspend fun <T> get(url: String, params: Map<String, Any>): Result<T>
}
// 生产实现：RetrofitNetworkClient
// 测试实现：MockNetworkClient（返回固定数据）
```

**可降级：**
- 网络失败时自动读取 Room 本地缓存
- 关键接口配置 CDN 备用域名
- 服务端配合实现接口熔断降级策略

### Q7：如何设计一个完整的 App 稳定性监控体系？

- **难度：** ⭐⭐⭐⭐⭐
- **考查点：** 稳定性治理、监控体系、系统设计

**参考答案：**

**监控体系分层：**

```
数据采集层：
  ├── Java/Kotlin Crash：UncaughtExceptionHandler
  ├── Native Crash：xCrash / Breakpad（捕获 tombstone）
  ├── Flutter Crash：FlutterError.onError + PlatformDispatcher.onError
  ├── ANR：FileObserver 监听 /data/anr 或 WatchDog 线程
  └── 自定义异常：业务 try-catch 手动上报

数据聚合层：
  ├── 按异常签名（类名+方法名+行号）去重聚合
  ├── 影响用户数、崩溃次数、首次/末次时间
  └── 关联设备信息（OS 版本、机型、内存）

告警策略：
  ├── 崩溃率 > 0.1% → P0 紧急告警（电话/短信）
  ├── 新增崩溃签名 → P1 告警（即时通讯）
  └── ANR 率 > 0.05% → P2 告警（工单）

响应机制：
  ├── 热修复通道（Tinker / Sophix）快速修复
  ├── 功能开关（Feature Flag）快速关闭问题模块
  └── 版本强更（关键 Bug 强制升级）
```

**关键指标：**

| 指标 | 健康阈值 | 说明 |
|------|---------|------|
| Crash 率 | < 0.1% | 崩溃 UV / 活跃 UV |
| ANR 率 | < 0.05% | ANR UV / 活跃 UV |
| 启动成功率 | > 99.5% | 启动未崩溃比例 |

### Q8：如何向团队推动引入一项新技术（如 Compose / Flutter）？完整的落地路径是什么？

- **难度：** ⭐⭐⭐⭐⭐
- **考查点：** 技术领导力、落地方法论、风险管理

**参考答案：**

**完整落地路径（以 Flutter 为例）：**

**阶段一：调研与验证（1-2 个月）**
1. 技术调研报告：原理、性能数据、社区成熟度、坑点盘点
2. 与团队现状对比：现有技术栈迁移成本、学习曲线
3. **小范围 POC**：选一个低风险新功能用 Flutter 实现，收集真实数据

**阶段二：立项与对齐（2 周）**
1. 将技术决策**量化为业务价值**："Flutter 开发效率提升 30%，双端维护成本减半"
2. 与产品/管理层对齐：不讲技术，讲人效和成本
3. 制定 **ADR（架构决策记录）**，记录决策背景、方案对比、取舍

**阶段三：试点落地（2-3 个月）**
1. 选择**业务隔离度高、重要性中等**的功能模块（风险可控）
2. 制定技术规范和最佳实践文档
3. 建立内部培训机制（学习小组 + 定期分享）

**阶段四：复盘推广**
1. 试点结束后复盘：性能数据、开发效率数据、Bug 率对比
2. 数据说话，决定是否全面推广
3. 建立标准模板和脚手架，降低后续接入成本

**加分点：** 提到"如果 POC 结论是负面的，要有勇气否定自己的预期，避免沉没成本效应推动错误技术决策"。

### Q9：如果你接手了一个技术债务严重的 Android 项目，如何在不停业的前提下逐步重构？

- **难度：** ⭐⭐⭐⭐⭐
- **考查点：** 技术债务管理、渐进式重构、工程领导力

**参考答案：**

**核心原则：** 绞杀者模式（Strangler Fig Pattern）——不推倒重来，而是新旧并存，逐步替换。

**执行步骤：**

**1. 现状摸底（不动代码）**
- 梳理架构现状图：模块依赖、层级违规、技术债务分布
- 识别高风险区域（高 Crash 率、高改动频率、强耦合）
- 建立基础测试覆盖率基线

**2. 建立防腐层**
- 在新旧代码之间建立 Facade 层，隔离腐化代码
- 新功能统一在防腐层外按新规范开发，不进入旧代码

**3. 优先级排序**
```
高影响 + 低改动成本 → 立即重构
高影响 + 高改动成本 → 分阶段规划
低影响 + 低改动成本 → 顺手优化
低影响 + 高改动成本 → 暂不处理
```

**4. 渐进式替换**
- 每个迭代预留 20% 时间用于技术债务偿还
- 每次重构前先补测试，重构后验证测试通过
- 小步快跑，每次 PR 只做一件事

**5. 防止债务反弹**
- 建立 CI Lint 规则，新代码不能违反架构约束
- Code Review 中严格把控分层规范
- 将"不新增技术债务"作为团队 OKR 指标

### Q10：跨平台方案（Flutter / React Native / 原生）如何进行技术选型？你的评估框架是什么？

- **难度：** ⭐⭐⭐⭐⭐
- **考查点：** 技术选型方法论、架构师综合判断力

**参考答案：**

**评估维度框架：**

| 维度 | 权重 | Flutter | React Native | 纯原生 |
|------|------|---------|-------------|--------|
| 渲染性能 | 25% | ⭐⭐⭐⭐⭐（自绘引擎） | ⭐⭐⭐（JS Bridge 开销） | ⭐⭐⭐⭐⭐ |
| 开发效率 | 20% | ⭐⭐⭐⭐（热重载+一套代码） | ⭐⭐⭐⭐ | ⭐⭐（双端各写）|
| 生态成熟度 | 15% | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 原生能力访问 | 15% | ⭐⭐⭐（需 Channel）| ⭐⭐⭐（需 Bridge）| ⭐⭐⭐⭐⭐ |
| 团队学习成本 | 10% | ⭐⭐⭐（学 Dart） | ⭐⭐⭐⭐（Web 背景友好）| ⭐⭐⭐⭐⭐ |
| 长期维护风险 | 15% | ⭐⭐⭐⭐（Google 强背书） | ⭐⭐⭐⭐（Meta）| ⭐⭐⭐⭐⭐ |

**决策树：**
```
业务以展示为主，重动画？→ Flutter
团队有大量 Web/JS 背景？→ React Native 优先考虑
需要深度原生能力（蓝牙/AR/支付硬件）？→ 原生 + Flutter 局部混入
团队全是 Android/iOS 原生？业务复杂？→ 原生起步，逐步引入 Flutter
```

**架构师答题要点：** 没有银弹，选型的核心是**匹配当前团队能力和业务阶段**，而非追求技术先进性。选型后要建立评估机制，定期回顾决策是否仍然有效。
