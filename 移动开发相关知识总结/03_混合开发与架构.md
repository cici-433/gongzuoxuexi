## 一、Flutter 混入 Android 项目（Add-to-App）

### 1.1 三种接入方式

| 方式 | 适用场景 | 说明 |
|------|---------|------|
| `FlutterActivity` | 整屏 Flutter 页面 | 最简单，直接 startActivity |
| `FlutterFragment` | Flutter 嵌入原生页面的一部分 | 灵活，可与原生 View 混排 |
| `FlutterView` | 极度自定义（少用） | 直接操作 View 层级 |

**接入步骤（FlutterFragment 方式）：**
```kotlin
// 1. 添加依赖（flutter module 作为 aar 或源码依赖）
// settings.gradle
include ':flutter_module'
setBinding(new Binding([gradle: this]))
evaluate(new File(settingsDir.parentFile, 'flutter_module/.android/include_flutter.groovy'))

// 2. 创建并显示 FlutterFragment
val flutterFragment = FlutterFragment
    .withNewEngine()
    .initialRoute("/home")
    .build<FlutterFragment>()

supportFragmentManager.beginTransaction()
    .add(R.id.fragment_container, flutterFragment, FLUTTER_FRAGMENT_TAG)
    .commit()
```

### 1.2 FlutterEngine 预热（关键性能优化）

**问题：** 首次创建 FlutterEngine 耗时约 300ms-1s，体验差。

**解决：提前预热引擎**

```kotlin
// Application.onCreate() 中提前初始化
class MyApp : Application() {
    lateinit var flutterEngine: FlutterEngine

    override fun onCreate() {
        super.onCreate()
        flutterEngine = FlutterEngine(this).apply {
            dartExecutor.executeDartEntrypoint(
                DartExecutor.DartEntrypoint.createDefault()
            )
        }
        // 注册到缓存，供后续复用
        FlutterEngineCache.getInstance().put("main_engine", flutterEngine)
    }
}

// 使用缓存引擎（0 冷启动开销）
FlutterFragment.withCachedEngine("main_engine").build<FlutterFragment>()
```

**Q：什么时候需要多引擎？**
> 多个相互独立的 Flutter 功能模块（如 IM 聊天页 + 商品详情页）可使用不同引擎，互不干扰状态。但每个引擎约占 4-6MB 内存，需控制数量，通常不超过 3 个。

## 二、混合通信机制

### 2.1 双向通信完整示例

**场景：** Flutter 页面调用 Android 获取 GPS 定位，Android 持续推送位置更新给 Flutter。

```dart
// Flutter 端
class LocationService {
  static const _method = MethodChannel('location/method');
  static const _event = EventChannel('location/stream');

  // 一次性获取
  Future<Map> getLastLocation() =>
      _method.invokeMapMethod('getLastLocation');

  // 持续监听
  Stream<Map> get locationStream =>
      _event.receiveBroadcastStream().cast<Map>();
}
```

```kotlin
// Android 端
class LocationPlugin(private val context: Context) {
    fun register(messenger: BinaryMessenger) {
        // MethodChannel
        MethodChannel(messenger, "location/method")
            .setMethodCallHandler { call, result ->
                when (call.method) {
                    "getLastLocation" -> result.success(getLastLocation())
                    else -> result.notImplemented()
                }
            }

        // EventChannel（持续推送）
        EventChannel(messenger, "location/stream")
            .setStreamHandler(object : EventChannel.StreamHandler {
                private var locationListener: LocationListener? = null
                override fun onListen(args: Any?, sink: EventChannel.EventSink) {
                    locationListener = LocationListener { location ->
                        sink.success(mapOf("lat" to location.latitude, "lng" to location.longitude))
                    }
                    locationManager.requestUpdates(locationListener!!)
                }
                override fun onCancel(args: Any?) {
                    locationManager.removeUpdates(locationListener!!)
                }
            })
    }
}
```

## 三、混合路由管理

### 3.1 问题根源

原生和 Flutter 各自维护独立的路由栈，混合使用时会产生：
- 返回手势不一致（原生 back 和 Flutter pop 冲突）
- 双栈管理复杂，页面生命周期混乱
- 内存中存在多个 Activity + Flutter 页面栈

### 3.2 方案对比

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **官方 Add-to-App** | 每次新建 FlutterActivity | 简单 | 多引擎内存大 |
| **FlutterBoost** | 统一路由容器，单引擎 | 内存小，栈统一 | 侵入性强 |
| **自研 Mix Stack** | 参考 FlutterBoost 思路 | 可定制 | 维护成本高 |

**FlutterBoost 核心思路：**
```
所有页面（原生 + Flutter）均通过统一 Router 调度
FlutterBoostActivity 作为容器，复用单一 FlutterEngine
每个"Flutter 页面"本质是 Flutter 内部路由的不同 route
原生页面通过 BoostNavigator.open("flutter://home") 跳转
```

**Q：你们项目如何解决混合路由问题？**
> （结合实际说，参考答法）我们采用单引擎 + 统一路由调度方案。所有跳转收口到 Router 模块，原生页面用 Intent，Flutter 页面通过 FlutterEngine 内部 Navigation。为避免双栈混乱，规定 Flutter 到原生跳转必须通过 MethodChannel 调用，不允许直接操作 Activity 栈，并在 NavigatorObserver 中同步路由状态以支持埋点。

## 四、大型 App 架构设计（架构师核心）

### 4.1 整体分层架构

```
┌─────────────────────────────────────────────────────┐
│                   App Shell（壳工程）                  │
├─────────────┬──────────────┬───────────────────────── ┤
│ Feature A   │  Feature B   │  Feature C (Flutter)     │  ← 功能层
│ (原生)      │  (原生)      │                          │
├─────────────┴──────────────┴──────────────────────────┤
│              Business Core（业务基础层）                │  ← 业务共享
│   用户中心 / 账号体系 / 埋点 / ABTest                   │
├───────────────────────────────────────────────────────┤
│              Infrastructure（基础设施层）               │  ← 技术底座
│   Network / Cache / DB / ImageLoader / Router         │
└───────────────────────────────────────────────────────┘
```

### 4.2 依赖注入体系（Hilt）

**Q：为什么用 Hilt 而不是手动单例？**
> 手动单例有几个问题：全局状态难以测试（单测需要 Mock 困难）；生命周期不受控（内存泄漏风险）；模块间隐式耦合。Hilt 提供编译时依赖图验证，测试时可替换 TestModule，生命周期与 Android 组件绑定，是架构级的改善。

### 4.3 网络层设计

```kotlin
// 统一结果封装
sealed class ApiResult<out T> {
    data class Success<T>(val data: T) : ApiResult<T>()
    data class Error(val code: Int, val message: String) : ApiResult<Nothing>()
    data object Loading : ApiResult<Nothing>()
}

// Repository 层统一处理
class UserRepository(private val api: UserApi) {
    suspend fun getUser(id: String): ApiResult<User> = try {
        val response = api.getUser(id)
        if (response.isSuccessful) {
            ApiResult.Success(response.body()!!)
        } else {
            ApiResult.Error(response.code(), response.message())
        }
    } catch (e: IOException) {
        ApiResult.Error(-1, "Network error")
    }
}
```

## 五、稳定性体系设计

### 5.1 Crash 监控分层

```
第一层（原生 Crash）：
  - Java Exception → Thread.UncaughtExceptionHandler
  - Native Crash    → Breakpad / xCrash 捕获 tombstone

第二层（Flutter Crash）：
  - FlutterError.onError = (details) { uploadToServer(details); }
  - PlatformDispatcher.instance.onError  (未捕获异常)

第三层（非崩溃异常）：
  - ANR 监控（FileObserver 监听 /data/anr/traces.txt）
  - 业务异常（try-catch + 上报）
```

### 5.2 APM 性能监控体系

```
指标采集：
├── 启动耗时（冷/热/温）
├── 页面加载耗时（onResume → 首帧）
├── 帧率（FPS / 卡顿帧占比）
├── 内存（PSS / Java Heap / Native Heap）
├── CPU 使用率
├── 网络（请求耗时 / 成功率 / 流量）
└── 电量（耗电量估算）

告警策略：
├── 启动耗时 P99 > 3s → 告警
├── 崩溃率 > 0.1% → 紧急告警
├── OOM 率 > 0.05% → 告警
└── 卡顿率 > 5% → 告警
```

**Q：如何设计一个无侵入的 APM SDK？**
> 核心是**字节码插桩（ASM/Transform）**：
> 1. 通过 Gradle Plugin 在编译期扫描字节码
> 2. 在关键方法（Activity.onCreate/onResume、网络请求）入口/出口插入耗时采集代码
> 3. 业务代码零修改，完全无感知
> 4. 对于 Native 层用 PLT Hook 拦截系统调用
> 5. 数据通过独立线程 + 批量上报，不影响主线程性能

## 六、技术选型方法论

### 6.1 跨平台方案评估矩阵

| 维度 | Flutter | React Native | 纯原生 |
|------|---------|-------------|--------|
| 渲染性能 | ⭐⭐⭐⭐⭐ 自绘，高帧率 | ⭐⭐⭐ JS Bridge 有开销 | ⭐⭐⭐⭐⭐ |
| 开发效率 | ⭐⭐⭐⭐ 热重载，一套代码 | ⭐⭐⭐⭐ | ⭐⭐ Android+iOS 各写一套 |
| 生态成熟度 | ⭐⭐⭐⭐ 快速成长 | ⭐⭐⭐⭐⭐ Meta 背书 | ⭐⭐⭐⭐⭐ |
| 接入成本 | ⭐⭐⭐ 需学 Dart | ⭐⭐⭐⭐ Web 开发者友好 | ⭐⭐⭐⭐⭐ |
| 包体积 | +5-10MB | +7-15MB | +0 |
| 原生能力访问 | 需 Platform Channel | 需 Native Module | 直接访问 |

**Q：公司要做跨平台，你如何做技术决策？**
> 我会从以下维度评估：
> 1. **业务特点**：纯展示类 → Flutter 优先；重度原生交互（相机/蓝牙）→ 考虑成本
> 2. **团队现状**：Android/iOS 开发者比例，学习曲线成本
> 3. **性能要求**：动画帧率敏感 → Flutter；快速迭代 → RN
> 4. **维护风险**：Flutter Google 背书，生态增速快；RN Meta 背书，生态更成熟
> 5. **落地策略**：不建议一次性全量迁移，以新功能模块试点验证，数据说话
