## 一、大型系统设计题（面试高频）

### 1.1 设计一个图片加载框架

**考察点：** 系统设计能力、对 Glide/Coil 原理的理解

**设计要点：**
```
请求层：
  - 统一 Request 对象（URL / 尺寸 / 变换 / 占位图）
  - 请求队列管理（优先级 / 并发控制）

缓存层（三级缓存）：
  - L1: 内存活跃缓存（WeakReference，与 GC 联动）
  - L2: 内存 LRU 缓存（固定大小，LruCache）
  - L3: 磁盘 LRU 缓存（DiskLruCache）

解码层：
  - BitmapFactory.Options.inSampleSize 采样压缩
  - BitmapPool 复用（inBitmap）
  - 根据 ImageView 尺寸按需解码

显示层：
  - 切回主线程设置 Bitmap
  - 动画过渡（CrossFade）
  - 生命周期绑定（页面销毁自动取消）
```

**Q：Glide 如何感知 Activity 生命周期？**
> Glide 在 Activity 中添加一个无 UI 的 `RequestManagerFragment`，通过 Fragment 的 `onStart/onStop/onDestroy` 回调驱动请求的暂停/恢复/取消，无需业务代码手动管理。

### 1.2 设计一个 Android 推送系统

**架构图：**
```
服务端
  └── 消息推送服务
        ├── 长连接管理（用户 ID → 连接 Channel 映射）
        └── 消息队列（Kafka）

客户端
  ├── 长连接保活方案
  │   ├── 心跳机制（APP 前台 30s / 后台 3min）
  │   ├── 进程保活（前台 Service）
  │   └── 系统通道（FCM / 厂商推送）兜底
  ├── 离线消息
  │   └── App 启动时拉取离线消息
  └── 消息可靠性
      ├── 消息 ID + 去重
      └── ACK 确认机制
```

**Q：如何保证消息的可靠到达？**
> 1. **QoS 机制**：类似 MQTT，支持 0（最多1次）/ 1（至少1次）/ 2（精确1次）
> 2. **客户端 ACK**：收到消息后回包 ACK，服务端未收到 ACK 则重推
> 3. **消息 ID 去重**：客户端维护已处理消息 ID 集合，避免重复处理
> 4. **离线补偿**：App 后台/离线期间消息入 MQ，上线后按序拉取

### 1.3 设计一个 App 灰度发布系统

```
灰度维度：
  ├── 用户维度（UID 取模 / 用户标签 / 白名单）
  ├── 设备维度（设备 ID / 机型 / 系统版本）
  └── 地域维度（省市 / 运营商）

控制台：
  ├── 实验配置（流量比例 / 目标用户群）
  └── 实时监控（崩溃率 / 转化率 / 留存率）

客户端 SDK：
  ├── 启动时拉取实验配置
  ├── 本地缓存（离线可用）
  └── 埋点上报实验曝光
```

**Q：灰度和 A/B 测试的区别？**
> 灰度是**风险控制**工具，目标是平稳发布新功能，发现问题快速回滚。A/B 测试是**决策工具**，目标是通过数据对比验证哪个方案更优，通常同时运行多个变体，需统计显著性分析。

## 二、Gradle 构建体系（工程化）

### 2.1 构建速度优化

| 手段 | 效果 |
|------|------|
| 开启构建缓存 `org.gradle.caching=true` | 复用已构建 task |
| 并行构建 `org.gradle.parallel=true` | 多模块并行编译 |
| 配置按需加载 `org.gradle.configureondemand=true` | 只配置相关模块 |
| 升级 AGP / Gradle 版本 | 持续有性能改进 |
| 减少注解处理器（KAPT → KSP） | KSP 比 KAPT 快 2-10x |
| 避免动态版本依赖 `implementation 'lib:+'` | 会每次网络请求 |

### 2.2 自定义 Gradle Plugin

**场景：** 统一各模块配置、自动插桩、打包检查

```kotlin
// buildSrc/src/main/kotlin/AppPlugin.kt
class AppPlugin : Plugin<Project> {
    override fun apply(target: Project) {
        target.plugins.apply("com.android.library")
        val android = target.extensions.getByType<LibraryExtension>()
        android.apply {
            compileSdk = 34
            defaultConfig { minSdk = 21 }
            buildTypes {
                release { isMinifyEnabled = true }
            }
        }
    }
}
```

## 三、团队技术治理

### 3.1 技术规范体系

**代码规范工具链：**
```
静态检测：
  - Kotlin: Detekt（自定义规则集）
  - Android: Lint（自定义 Lint rule）

格式化：
  - ktfmt / ktlint（统一代码风格）

提交规范：
  - Conventional Commits（feat/fix/refactor/perf）
  - Git hooks（pre-commit 跑 lint，commit-msg 校验格式）

CI 强制检查：
  - PR 合并前必须通过所有 Lint 检查
  - 测试覆盖率不低于基线
```

### 3.2 Code Review 机制

**有效 CR 的原则：**
1. **目标明确**：安全性 > 正确性 > 可维护性 > 性能 > 风格
2. **及时**：PR 提交后 24h 内完成 Review
3. **建设性**：提建议而非命令，说明"为什么"
4. **分级**：Blocking（必须改）/ Suggestion（建议改）/ Nit（小问题）

**架构 Review 关注点（区别于普通 CR）：**
- 是否符合分层原则（依赖方向是否正确）
- 是否有跨层直接调用
- 公共接口设计是否合理（向后兼容性）
- 是否引入新的循环依赖

### 3.3 技术债务管理

**技术债务四象限：**
```
              高影响
                │
      优先偿还   │   规划偿还
  ─────────────────────────── 难以偿还
      接受存在   │   观察跟踪
                │
              低影响
```

**债务管理流程：**
1. **识别**：架构评审、Code Review、定期技术债务 Backlog Review
2. **量化**：用时间估算（修复该债务需要多少工时）
3. **优先级**：按影响 × 修复成本排序
4. **偿还**：每个迭代预留 20% 时间用于技术债务偿还
5. **预防**：通过规范 + 工具降低新债务产生速度

**Q：你如何向产品/管理层争取技术重构的资源？**
> 关键是**量化收益**，避免只谈技术：
> - "当前代码结构导致每个需求平均多出 2 天开发时间，重构后预计节省 40%"
> - "该模块 Crash 率 0.3%，影响每月 XX 用户，重构后预计降至 0.05%"
> - 不是"代码写得不好"，而是"技术问题带来的业务成本"

## 四、面试高频系统设计题汇总

### Q1：如何设计一套 App 全局异常处理机制？

```kotlin
// 全局未捕获异常（Java 层）
Thread.setDefaultUncaughtExceptionHandler { thread, throwable ->
    // 1. 收集现场信息（设备信息/内存/线程栈）
    // 2. 本地持久化（下次启动上报，避免网络还没初始化）
    // 3. 根据策略决定是否重启 App
    ErrorReporter.save(throwable)
    restartApp() // 或者直接 System.exit
}

// Flutter 层
FlutterError.onError = (FlutterErrorDetails details) {
    Zone.current.handleUncaughtError(details.exception, details.stack!);
};
PlatformDispatcher.instance.onError = (error, stack) {
    ErrorReporter.report(error, stack);
    return true; // 返回 true 表示已处理
};
```

### Q2：如何设计一个支持离线的 App 数据同步方案？

```
策略：
1. 本地优先（Offline-First）
   - 所有操作先写本地 Room DB
   - 后台 WorkManager 任务同步到服务端
   - 冲突解决：Last-Write-Win 或 CRDTs（复杂场景）

2. 同步状态标记
   - 每条记录有 syncStatus: PENDING / SYNCED / FAILED
   - 网络恢复时重试 PENDING 记录

3. 乐观更新 UI
   - 操作立即反映在 UI，不等待网络返回
   - 服务端返回错误时回滚本地状态
```

### Q3：如何设计多端（Android/iOS/Flutter）统一的埋点 SDK？

```
设计要点：
1. 统一数据模型
   - EventName + Properties(Map) + CommonProperties(设备/用户)

2. 采集方式
   - 代码埋点（精确，维护成本高）
   - 自动化埋点（页面曝光/点击，无代码/可视化圈选）
   - 混合方案（关键转化手动埋点 + 通用行为自动化）

3. 可靠性
   - 本地缓存 + 批量上报（避免频繁网络请求）
   - 失败重试（指数退避）
   - 离线缓存（App 后台/离线期间本地存储）

4. 性能
   - 子线程采集和上报，不阻塞主线程
   - 数据压缩（gzip/protobuf）

5. 隐私合规
   - GDPR/PIPL：用户授权前不采集个人数据
   - 数据脱敏（手机号/身份证 mask）
```

## 五、架构师面试答题框架

### STAR 法则应用

面试官问**项目经验**类题目时，用 STAR 结构：
- **S**ituation（背景）：项目规模、团队状况、面临的挑战
- **T**ask（任务）：你的职责是什么
- **A**ction（行动）：你做了什么决策，为什么这样做，有哪些取舍
- **R**esult（结果）：量化数据，学到了什么

### 系统设计题答题框架

1. **澄清需求**（2min）：功能范围、规模估算（DAU/QPS）、约束条件
2. **高层设计**（5min）：画出模块图，说明各组件职责
3. **深入核心**（10min）：重点展开1-2个最复杂的子系统
4. **权衡取舍**（3min）：主动说出方案的局限性和可改进点
5. **演进路线**（2min）：如何从 MVP 演进到大规模方案

### 体现架构师视野的关键词

| 普通答法 | 架构师答法 |
|---------|-----------|
| "我们用了 Retrofit" | "我们在 Retrofit 基础上封装了统一的错误处理、鉴权刷新、降级策略" |
| "用 Glide 加载图片" | "我们统一了图片加载门面，底层可替换为任意实现，并加入了 CDN 加速策略" |
| "用 Room 存数据" | "我们的数据层遵循 Repository 模式，Room 作为本地数据源之一，支持降级到内存缓存" |
| "发现内存泄漏修了" | "我们建立了内存治理体系：线下 LeakCanary，线上 PSS 监控，定期 Heap Dump 分析" |
