# 移动端 App 核心关注清单（架构/负责人视角）

> 目标：把“怎么做 App”讲成面试可口述的结构化答案。建议按 “一句话原则 → 关键指标 → 核心抓手 → 风险兜底” 的顺序回答。

## 30 秒总览（面试口述版）
- 目标与业务：围绕业务目标拆解端侧 KPI，需求必须可观测、可验收  
- 体验：关键链路优先（首屏/登录/搜索/下单），一致性与可访问性长期建设  
- 底座（稳定/性能/合规）：启动与流畅、包体与耗电、Crash/ANR；数据最小化与加密传输存储  
- 架构与工程化：模块化边界与单向依赖、状态可预测；灰度/开关/回滚 + CI/CD/测试/监控  
- 数据闭环：埋点 Schema 与质量对账，AB + Guardrail 指标保证优化可控

## 十年+的核心能力与价值（区别于普通开发）
### 面试表达框架
- 一句话原则：普通开发交付功能，高级工程师交付“结果 + 体系 + 团队能力”
- 关键证据：用指标与机制证明（提升了什么、降低了什么、建立了什么）
- 风险兜底：能控线上风险（灰度/回滚/告警/应急），把 MTTR 降下来

### 核心能力清单（抓住 5-7 条即可）
- 结果导向：把需求翻译成指标、链路、实验与复盘，交付可验证的业务增量
- 体系化治理：把稳定性/性能/包体/耗电做成指标体系、门禁与闭环，让质量自动发生
- 架构演进：在历史包袱与资源约束下做边界与依赖治理，保证可持续迭代不失控
- 复杂问题定位：用分阶段打点与分桶分析把问题定位到模块/阶段/机型/网络，形成可回归修复
- 线上治理：统一开关、灰度、回滚与监控告警，降低事故概率并控制损失面
- 跨团队推进：拉齐 PM/设计/后端/数据目标与节奏，推动关键项目落地
- 团队提效：Owner 制、评审文化、规范与复用体系，把个人能力沉淀成团队能力

## 公司通常更重视什么（评价维度）
- 业务结果：关键指标提升、核心链路转化提升、版本按时交付与可控迭代
- 质量底线：Crash/ANR、启动/流畅回退次数、线上事故次数、回滚率、MTTR
- 研发效率：发版频次、CI 通过率、关键路径测试覆盖、回归成本、缺陷密度
- 技术影响力：架构治理带来的长期收益（模块化、统一组件/网络/埋点、降本增效）
- 组织贡献：培养与梯队、流程机制与知识沉淀、关键人风险降低

## 口述模板（可直接背）
- 我通常从 5 个方面看一个 App：目标与业务、体验、底座（稳定/性能/合规）、架构与工程化、数据闭环
- 我做事的方法是：先定义指标与口径 → 建立观测与门禁 → 灰度验证 → 上线复盘形成机制
- 我能带来的价值是：不只做功能，还能把风险控住、把质量体系建起来、把团队交付能力做出来

## 目标与业务
### 面试表达框架
- 一句话原则：业务指标先定，端侧只做可观测、可验收的优化
- 关键指标：留存/转化/频次/ARPU + 端侧体验/稳定性指标（作为守护指标）
- 核心抓手：需求评审绑定指标与埋点 → 灰度/AB → 上线复盘
- 风险兜底：口径统一、分层分桶（机型/网络/地域）看分布与异常

### 常用指标（举例即可）
- 业务类：次日/7 日留存、注册转化率、首购转化率、成功下单率、DAU/WAU
- 体验类：首屏 TTI、接口 TTFB、关键路径成功率、超时/重试率

### 可执行动作
- OKR → 端侧 KPI → 仪表盘 → 周报/复盘，需求从评审开始就带“验收口径”
- 实验框架：灰度/AB + Guardrail（Crash/ANR/启动/掉帧）不退化

### 常见陷阱
- 目标不可观测/口径不一致；只看均值不看 P95/P99；只看总量不分机型/网络

## 用户体验
### 面试表达框架
- 一句话原则：关键链路不卡、信息可理解、交互可预测
- 关键指标：关键路径成功率、首屏 TTI、页面渲染耗时、错误/超时率、用户反馈
- 核心抓手：骨架与占位、渐进加载、弱网降级、体验一致性建设
- 风险兜底：频控与可回退，避免“体验提升带来打扰或不稳定”

### 关键链路（面试说 1-2 条即可）
- 首屏：骨架屏先上、首屏数据分“必需/非必需”，非必需延后
- 登录/支付：错误可解释、可重试、可恢复，避免用户死路

### 一致性与可用性
- 设计系统：组件库 + Token（颜色/字号/间距/圆角）统一，减少“每页一个样”
- 可访问性：语义标签、可聚焦、触达面积（≥44pt）、对比度
- 国际化：语言包版本化、时间/货币/度量单位本地化、RTL 验收

### 通知与打扰治理
- 策略：频控/去重、静默时段、分群、偏好中心
- 验收：到达-点击-转化漏斗、退订率、投诉率

## 性能与稳定性
### 面试表达框架
- 一句话原则：稳定优先，其次性能；性能用指标说话，优化必须可回归
- 关键指标：启动（TTI）、流畅（帧耗/Jank）、包体、耗电、Crash/ANR
- 核心抓手：分阶段打点定位瓶颈 → 门禁（PR/发版）→ Top 问题闭环
- 风险兜底：灰度 + 回滚 + 监控告警，避免线上回退成本过高

### 启动与流畅
- 指标：冷/暖/热启动 TTI、P95/P99 帧耗、Jank 率、慢启动占比
- 抓手：按需初始化、减少主线程同步、图片按需解码、列表优化（itemExtent/复用/边界）

### 体积与资源
- 指标：包体总量与增量、资源体积占比、启动期资源加载量
- 抓手：资源清理、字体子集化、分 ABI、按需加载与压缩

### 耗电与后台
- 指标：典型场景耗电曲线、后台唤醒频次、后台存活与打断原因
- 抓手：合并定时器、降低唤醒频率、平台合规的后台任务策略

### 稳定性
- 指标：Crash Free、ANR、Top Crash 分布、回归率
- 抓手：统一异常捕获与符号化、Top Crash 周会、分配与 SLA

### 缓存与离线
- 指标：缓存命中率/回源率、弱网成功率、离线可用率
- 抓手：多级缓存、过期/回收策略、弱网降级与离线兜底

## 安全与合规
### 面试表达框架
- 一句话原则：数据最小化与可告知，传输/存储全链路保护，第三方可控可审计
- 关键指标：权限授权率/拒绝率、合规审计通过率、敏感数据泄漏事件为 0
- 核心抓手：数据清单 + SDK 准入 + 权限网关 + 加密与证书策略
- 风险兜底：Pinning 滚更策略、灰度开关、应急下架与替换预案

### 落地清单
- 数据合规：数据目录/最小化、敏感权限前置说明、隐私政策可追溯
- 传输存储：强制 HTTPS、证书校验/Pinning、本地敏感数据加密（KeyStore/Keychain）
- SDK 治理：准入评审、版本白名单、权限最小化、定期审计

## 架构与可演进
### 面试表达框架
- 一句话原则：边界清晰、依赖单向、状态可预测；以“可替换/可测试/可灰度”为演进目标
- 关键指标：模块耦合度、发布频率、缺陷密度、回滚耗时、复用率（跨端）
- 核心抓手：模块化 + 依赖治理 + 状态管理规范 + Feature Flag
- 风险兜底：接口稳定、第三方适配层、关键能力可降级/可回滚

### 落地清单
- 模块化：按业务域/特性拆包，维护依赖图；模块可独立构建/测试
- 状态管理：单向数据流、明确副作用边界；大功能有可回放的事件链路
- 配置开关：开关中心、分流策略（用户/地域/版本），灰度与回滚标准化
- 可替换性：第三方/平台能力封装适配层，保证替换成本可控
- 跨平台：Channel 接口规范化，复用率与性能差异可量化

## 数据与分析
### 面试表达框架
- 一句话原则：先把数据口径与质量做对，再谈增长与优化
- 关键指标：埋点覆盖率、丢失率、延迟、口径一致性、实验显著性
- 核心抓手：Schema 管理 + 质量对账 + AB 体系 + 仪表盘复盘
- 风险兜底：Guardrail 指标（稳定/性能）不退化，避免“增长但体验变差”

### 落地清单
- 埋点：命名规范、Schema 评审、关键路径覆盖
- 质量：丢失率/延迟/去重，端云抽样对账与告警
- 实验：样本量与显著性校验，守护指标不退化
- 复盘：按业务域看板，优化任务入 Backlog 并跟踪闭环

## 平台与端能力
### 面试表达框架
- 一句话原则：把“系统差异与约束”做成统一能力，业务只关心稳定的接口
- 关键指标：深链成功率、前后台恢复成功率、权限拒绝率、推送转化漏斗
- 核心抓手：路由守卫、权限网关、任务恢复、统一日志与降级
- 风险兜底：参数校验、异常兜底与用户引导、可回退策略

### 落地清单
- 路由/深链：统一解析与守卫，冷/热启动差异处理，参数校验
- 前后台：任务保存/恢复，节电策略适配，一致性校验
- 权限/硬件：权限网关、降级策略、异常日志与用户指引
- 推送：渠道对比、模板化、落地页参数追踪、反作弊

## 工程效率与质量
### 面试表达框架
- 一句话原则：让质量“自动发生”，用工程化降低人为失误
- 关键指标：构建成功率、回归耗时、线上缺陷率、回滚耗时、发版频次
- 核心抓手：CI 门禁 + 自动化测试 + 标准化发版 + 评审规范
- 风险兜底：主干保护、灰度发布、应急回滚与演练

### 落地清单
- 基线：lint/format/依赖审计/安全扫描进 CI，PR 过门禁才可合入
- 测试：关键路径 E2E 常绿 + 冒烟用例合入触发 + 真机/云测并行
- CI/CD：多环境构建、版本号规范、灰度分批与一键回滚
- 评审：覆盖率/复杂度/包体增量门禁，主干开发与分支策略
- 文档：变更附 ADR，关键系统 Runbook，Onboarding 可复制

## 发布与运营
### 面试表达框架
- 一句话原则：发版是系统工程，靠节奏、灰度、监控与回滚把风险降到最低
- 关键指标：发版成功率、回滚率、线上故障 MTTR、审核通过率
- 核心抓手：上架清单、Release Train、分级告警、值班机制
- 风险兜底：紧急通道（热修/回滚/黑名单）、演练与复盘

### 落地清单
- 商店合规：上架清单检查表、预审流程、区域差异策略
- 版本策略：固定节奏 + 灰度 + 黑名单，紧急修复通道
- 监控告警：阈值分级、值班与响应手册、演练复盘
- 客诉反馈：渠道聚合、崩溃/日志关联、快速修复闭环

## 团队与治理
### 面试表达框架
- 一句话原则：团队产能来自“明确责任 + 可预期流程 + 可复制能力”
- 关键指标：交付周期、缺陷率、人员梯队、技术债偿还比例、稳定性 KPI 达标率
- 核心抓手：Owner 制、OKR/优先级机制、知识沉淀与培养体系
- 风险兜底：容量评估与缓冲、关键人风险、平台变更预案

### 落地清单
- Owner：职责清晰、接口人明确、跨团队协作机制（例会/同步）
- OKR：季度 OKR、公示优先级、阻塞项管理
- 成长：双周分享、专项培训、代码阅读与评审文化、Mentor
- 计划：Roadmap 公示、容量评估与缓冲、技术债台账与偿还计划

## 优先级建议（落地抓手）
- 建“可观测三板斧”：稳定性（Crash/ANR）、性能（启动/掉帧）、体积预算  
- 建“能力开关 + 灰度 + 回滚”联动链，保障线上安全发布  
- 用“模块化 + 单向依赖 + 状态管理”稳住架构可演进与可测试  
- 关键路径金用例全自动化；新增需求必配埋点与评估机制  
- 敏感权限和数据采集最小化与显式告知，降低合规风险  
- 以跨端复用率与端侧 KPI 改善作为评审与技术决策硬指标  

## 术语口径（面试不展开也要会说）
- TTI：Time To Interactive，从启动到“首屏可稳定响应交互”的时间
- TTFB：Time To First Byte，从发起请求到收到响应首字节的时间
