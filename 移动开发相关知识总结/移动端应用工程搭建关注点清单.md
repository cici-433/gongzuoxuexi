# 移动端应用工程搭建关注点清单

> 目标：把工程底座设计到“可扩展、可维护、可发布”，降低后续返工与线上风险。

## 一、分层架构 + 模块化（可扩展）

重点在于：加新功能成本低

- **纵向分层（职责解耦）**：
  - **UI 层**（页面与视图组件）：只负责数据展示与用户交互，严禁在 Activity/ViewController/Widget 中混入业务计算或网络请求。
  - **业务逻辑层**（ViewModel/Presenter/UseCase）：负责状态流转与核心业务规则，充当 UI 与数据的桥梁。
  - **数据层**（Repository/DataSource）：负责提供纯净的数据模型，屏蔽数据来源（是从网络拉取还是本地缓存）。
  - **基础设施层**：提供纯技术能力（网络通道、数据库引擎、日志系统等）。
  - **总结**：通过隔离变化，让某一层在改变演进时，其他层能零成本复用。
    - UI层与业务层解耦，同一业务逻辑可以适配不同UI展现
    - 数据层与业务层解耦，同一业务可以适配不用的数据来源（网络/本地缓存）
    - 基础设施层屏蔽，基础设施的升级/替换，完全不影响上层业务（比如图片库、网络库）
- **横向模块化（业务隔离）**：
  - **业务模块**：按业务域拆分（如 `login`、`mine`、`home`、`pay` 等），模块间禁止反向依赖或循环依赖，通过路由或接口层（Service/Protocol）通信。
  - **基础组件**：将通用能力下沉为独立组件（如 `network`、`router`、`ui_components`），供各业务模块复用。
  - **总结**：通过模块化，本质是把系统拆成一组 边界清晰、依赖单向、可插拔 的业务单元，让“新增/替换/组合”都变成 局部改动 而不是“全局牵一发而动全身”。
    - 加新业务，不需要去改一堆旧模块内部代码
    - 基础能力下沉复用，不用每个模块重复造轮子，替换实现也不影响上层业务
    - 模块边界明确后，并行开发不互相阻塞
- **架构模式统一**：
  - 选定一种主架构并在全工程强制执行，避免多种模式混用带来的理解与协作成本。
  - 参考规范：Android 推荐 `MVVM + Jetpack`，iOS 推荐 `MVVM + Combine`，跨平台推荐 `Flutter 分层 + Repository`。
  - **总结**：通过统一架构模式，让新增功能的路径标准化，让扩展变成“按模板增量复制/组装”
- **终极目标**：避免代码全部堆砌在“巨石应用”中，确保在新增业务或替换底层技术（如换网络库）时，只需局部改动，保障工程边界与依赖方向长期稳定。

## 二、统一基础框架 + 规范（可维护）

重点在于：改现有东西低成本、低风险

- **基础能力收口（防重复造轮子）**：
  - **网络体系**：统一封装请求/响应模型、公共 Header 注入、全局防重试与超时控制、Token 自动刷新机制、统一错误码拦截与降级处理。
  - **路由体系**：建立中心化页面注册与跳转机制，支持携带复杂参数、统一拦截器（登录态校验、权限校验）、防重连点控制，以及跨端深链（DeepLink）分发。
  - **存储体系**：分层管理本地数据。结构化数据走数据库（如 Room/CoreData），轻量 KV 走高频缓存（如 MMKV），敏感数据走安全区（Keychain/Keystore）。
  - **媒体与资源**：图片加载统一收口，支持全局占位图、加载失败图、内存与磁盘缓存策略、图片圆角及高斯模糊等公共特效。
  - **总结**：把网络、路由、存储、图片等通用能力统一封装收口为基础组件，业务只面向统一接口复用，从而减少重复造轮子并提升一致性、稳定性与可维护性
- **规范与约束（防代码腐化）**：
  - **代码风格门禁**：强制推行 Lint/Format 规则，集成至提交前置检查（Pre-commit hook），拦截不合规命名与魔法数字。
  - **异常兜底机制**：全局捕获未处理异常，严禁业务代码中随意使用空的 `try-catch` 吞没错误，必须将异常抛出或记录到监控系统。
  - **设计系统（Design System）落地**：与 UI 团队对齐并抽象基础组件（Button、Toast、Dialog、骨架屏），统一全局色板、字号阶梯、间距规范，并内置深色模式/多尺寸机型适配策略。
  - **总结**：通过统一的代码风格门禁、异常处理规范与设计系统组件约束，把质量规则前置并制度化，持续抑制代码腐化，保证长期一致性与可维护性。
- **外部依赖治理**：
  - **SDK 统一初始化**：所有第三方 SDK（推送、分享、支付、统计）集中管理初始化时机，避免散落各处拖慢启动速度。
  - **合规与权限**：权限申请统一封装弹窗与说明逻辑，确保符合隐私合规红线，支持特定渠道/市场的 SDK 动态剥离。
  - **总结**：通过集中化管理第三方 SDK 的接入与初始化，并统一权限与合规处理策略，实现依赖可控、启动可优化、渠道可裁剪，从而降低合规与性能风险。
- **测试与可测性设计**：接口 Mock、单元测试结构、调试工具面板（切换环境/查看请求/清空缓存）、抓包开关
- **协作契约**：明确项目文档（环境搭建、打包指南），规范 Git 分支模型（如 GitFlow）、Commit 格式验证以及 Code Review 流程。

## 三、监控 + 安全 + 构建（可上线/可观测）

重点在于：保证线上稳定、风险可控、交付高效

- **全方位可观测性（线上“天眼”）**：
  - **稳定性监控**：接入崩溃收集平台，确保 Crash、ANR（应用无响应）、OOM（内存溢出）等致命错误能自动上报，并附带符号表/混淆映射还原能力。
  - **性能指标追踪**：监控核心性能数据，包括 App 启动耗时、首屏渲染时长、核心接口成功率/耗时分布、滑动卡顿率等。
  - **业务行为洞察**：统一埋点 SDK 封装，支持页面曝光、点击、自定义事件的规范化上报，保障用户行为路径可回溯。
- **多维安全防御（守住红线）**：
  - **通信与数据安全**：核心接口强制 HTTPS 加双向证书校验（防抓包），关键请求参数与返回结果加密，确保敏感信息在网络链路和本地存储中均不明文裸奔。
  - **端侧防护**：基础的防调试、防反编译、反注入配置，提高被恶意篡改或破解的门槛。
- **工程化构建与 CI/CD（解放生产力）**：
  - **多环境敏捷切换**：工程内置 Dev/Test/Pre/Prod 多环境配置，可一键或自动切换域名、埋点上报环境、调试开关等。
  - **自动化流水线**：搭建 CI/CD 流程，实现代码合并自动触发 Lint/单测，打通渠道包生成、代码加固、签名打包、资源混淆以及构建产物分发的全自动化闭环，并将构建结果与产物链接自动通知企微/飞书等协作工具。
  - **构建性能调优**：通过配置编译缓存、按需加载模块、裁剪不必要插件等手段，将本地增量编译和远端打包耗时压至最低，提升团队迭代效率。

## 四、工程目录结构（示例）

> 目标：让依赖方向天然正确，业务可插拔，基础能力可复用。

```text
repo-root/
  app/                          # 壳工程：Application、入口 Activity、全局组装(依赖注入/路由注册)、多环境配置
  core/                         # 基础组件（纯技术能力，供所有模块复用）
    network/                    # 网络封装：OkHttp/Retrofit、拦截器、统一错误模型、序列化
    router/                     # 路由/DeepLink：页面注册、跳转、拦截器、参数编解码
    ui_components/              # 设计系统组件：Button/Dialog/Toast/Loading 等
    logger/                     # 日志与埋点基础能力（可选）
    common/                     # 通用工具（尽量薄，避免变成“新巨石”）
  domain/                       # 领域层（稳定契约）：实体、UseCase、Repository 接口
    user/                       # 用户领域：User、UserRepository、GetUserUseCase...
    auth/                       # 登录态/鉴权领域：SessionStore、AuthRepository...
  data/                         # 数据实现层：domain 接口的实现（网络/DB/缓存）
    user/                       # UserRepositoryImpl、UserRemoteDataSource、UserLocalDataSource
    auth/                       # Token 刷新、存储实现等
  feature/                      # 业务模块（按业务域拆分，可独立开发、可插拔）
    login/                      # 登录相关页面与流程（依赖 domain/auth + domain/user）
    home/                       # 首页（依赖 domain/user 等）
    mine/                       # 我的（依赖 domain/user 等）
    pay/                        # 支付（依赖 domain/order 或 domain/pay 等）
  build-logic/                  # Gradle 约定插件/依赖版本管理（可选：buildSrc 或 convention plugins）
  config/                       # 工程规范与脚本：lint/ktlint/detekt、CI、代码扫描、签名配置模板（不放密钥）
  gradle/                       # Gradle wrapper/版本目录
  docs/                          # 项目文档（可选，按需）
```

### 依赖方向（推荐约束）

```text
feature:*  ->  domain:*  <-  data:*
feature:*  ->  core:* 
app        ->  feature:* + data:* + core:*   # 只在 app 做“组装”，避免业务互相依赖
```

### 单个模块内部（常见 package 组织）

```text
feature/home/src/main/java/.../
  ui/                           # Compose/View + 状态渲染
  vm/                           # ViewModel（或 presenter）
  navigation/                   # 路由入口、页面声明（对外暴露最小入口）
  di/                           # 仅本模块的依赖注入声明
  model/                        # UI Model（展示模型，避免直接使用 domain/entity）
```

## 一句话总结（Leader 视角）

- 分层架构 + 模块化（可扩展）
- 统一基础框架 + 规范（可维护）
- 监控 + 安全 + 构建（可上线）
